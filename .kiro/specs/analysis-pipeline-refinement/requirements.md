# Requirements Document

## Introduction

This feature refines the GhostFix analysis pipeline to produce more concise, actionable output. The current pipeline generates verbose results across scoring, research narrative, and issue diagnosis. The refinement focuses on condensing analysis output, prioritizing high-impact insights, and removing redundant or low-signal information so that the downstream repair agent and end users receive a tighter, more focused diagnosis.

## Glossary

- **Pipeline**: The sequence of processing stages (Crawler, Answer_Collector, Scoring_Engine, Research, Repair_Agent) that produce an AnalysisResult
- **Scoring_Engine**: The module that applies a 5-dimension rubric to crawl signals and citations, producing numeric scores, reasons, and issues
- **Research_Module**: The module that auto-discovers prompts and competitors, then synthesizes a narrative report from analysis findings
- **Narrative**: The markdown report generated by the Research_Module summarizing visibility gaps and recommended actions
- **Issue**: A diagnosed problem with a title, severity, dimension, and explanation (the `why` field)
- **DimensionScore**: A subscore for one of the five rubric dimensions including a reasons array
- **Answer_Collector**: The module that queries AI engines to track brand/competitor citation frequency
- **Concise_Output**: Analysis output that contains only high-signal findings, limited to the most impactful items, with short explanations
- **Refinement_Summary**: A new condensed top-level field on the AnalysisResult that provides a 2-3 sentence executive summary of the diagnosis

## Requirements

### Requirement 1: Limit Issue Output to Top Actionable Items

**User Story:** As a brand owner, I want the analysis to surface only the most impactful issues, so that I can focus on fixes that move the needle rather than reading through noise.

#### Acceptance Criteria

1. THE Scoring_Engine SHALL return a maximum of 3 issues in the `issues` array, sorted by severity (high first) then by gap ratio (lowest score/max first).
2. WHEN two issues share the same severity and the same score-to-max ratio, THE Scoring_Engine SHALL break the tie by the dimension's natural order as defined in the DIMENSION_MAX record (share_of_answer first, freshness_trust last).
3. THE Scoring_Engine SHALL first filter to include only issues where the score-to-max ratio is below 0.67, then cap the result at 3 items.
4. IF fewer than 3 issues have a score-to-max ratio below 0.67, THEN THE Scoring_Engine SHALL return only those qualifying issues without padding.

### Requirement 2: Generate an Executive Summary

**User Story:** As a brand owner, I want a short, plain-language summary of the overall diagnosis, so that I can understand the key finding without reading the full breakdown.

#### Acceptance Criteria

1. WHEN the analysis completes successfully, THE Pipeline SHALL produce a `summary` field on the AnalysisResult containing a Refinement_Summary as a plain-text string.
2. THE Refinement_Summary SHALL be between 1 and 3 sentences and no longer than 280 characters including whitespace.
3. THE Refinement_Summary SHALL include the numeric overall score (out of 100), the title of the highest-severity issue (selecting the first by dimension order when multiple issues share the same severity), and the competitor domain used in the analysis.
4. IF the analysis produces zero issues, THEN THE Pipeline SHALL generate a Refinement_Summary that includes the overall score and the competitor domain but omits the issue title.

### Requirement 3: Shorten the Research Narrative

**User Story:** As a brand owner, I want the research narrative to be concise and structured, so that I get the key competitive insights without a lengthy essay.

#### Acceptance Criteria

1. THE Research_Module SHALL produce a narrative containing no more than 200 whitespace-delimited words when generating findings.
2. THE Research_Module SHALL structure the narrative with exactly 3 sections in the following order: a headline finding of at most 1 sentence (maximum 30 words), a bullet list of the top 2 gaps ranked by issue severity (high before medium before low), and a recommended first fix of at most 1 sentence (maximum 30 words).
3. WHEN the LLM generates a narrative exceeding 200 words, THE Research_Module SHALL re-prompt the LLM with an instruction to condense the response to at most 200 words while preserving the 3-section structure, up to a maximum of 2 retry attempts.
4. IF the LLM fails to produce a narrative within 200 words after 2 retry attempts, THEN THE Research_Module SHALL truncate the narrative to the first 200 whitespace-delimited words, append an ellipsis character (…), and retain whatever complete sections fit within that limit.
5. IF the LLM returns an empty response or the generation call fails, THEN THE Research_Module SHALL return a fallback narrative stating the brand's score, the top-scoring competitor domain, and a message indicating that detailed findings are unavailable.

### Requirement 4: Deduplicate Citation Sources

**User Story:** As a brand owner, I want citation sources to be deduplicated, so that the same domain does not appear multiple times inflating the source list.

#### Acceptance Criteria

1. THE Answer_Collector SHALL deduplicate sources by domain (case-insensitive, with "www." prefix stripped), retaining only the first URL encountered per domain in insertion order across runs for a given prompt.
2. THE Answer_Collector SHALL return at most 5 unique-domain sources per prompt.
3. WHEN a source domain appears across multiple runs for the same prompt, THE Answer_Collector SHALL increment brandCitedCount or competitorCitedCount at most once per run in which the domain appears, not once per individual citation within that run.
4. IF fewer than 5 unique domains are collected across all runs for a prompt, THEN THE Answer_Collector SHALL return only the available unique-domain sources without padding or error.

### Requirement 5: Remove Low-Signal Crawl Fields from Output

**User Story:** As a brand owner, I want the analysis output to exclude signals that do not contribute to the scoring, so that the result payload stays small and focused.

#### Acceptance Criteria

1. THE Pipeline SHALL exclude the `pricingClarity` field from all CrawlSignals objects included in the final AnalysisResult response payload.
2. IF the `lastUpdated` field value is null in a CrawlSignals object, THEN THE Pipeline SHALL omit the `lastUpdated` field from that CrawlSignals object in the AnalysisResult response payload.
3. IF the `lastUpdated` field value is non-null in a CrawlSignals object, THEN THE Pipeline SHALL include the `lastUpdated` field with its value in that CrawlSignals object in the AnalysisResult response payload.
4. THE Pipeline SHALL pass the complete, unfiltered CrawlSignals (including `pricingClarity` and `lastUpdated`) to the scoring engine before applying any response-level field exclusions.

### Requirement 6: Provide a Configurable Verbosity Level

**User Story:** As an API consumer, I want to request either a concise or detailed analysis, so that I can choose the level of detail appropriate for my use case.

#### Acceptance Criteria

1. WHEN the `verbosity` field in AnalyzeRequest is set to "concise" or is omitted, THE Pipeline SHALL apply all refinement constraints from Requirements 1-5 to the response and return the refined AnalysisResult.
2. WHEN the `verbosity` field is set to "detailed", THE Pipeline SHALL return the AnalysisResult without applying the refinement constraints from Requirements 1-5, preserving all raw scores, full issue descriptions, and untruncated citation data.
3. IF the `verbosity` field is provided with a value other than "concise" or "detailed", THEN THE Pipeline SHALL reject the request with an error response indicating the invalid verbosity value and listing the accepted values.
4. THE Pipeline SHALL default to "concise" verbosity when the `verbosity` field is not provided by the caller.
